# 信息科技学生学习综合门户平台设计文档

日期：2026-03-25

## 1. 当前项目判断

- 这不是一个通用 LMS，而是一个围绕“信息科技课程课堂教学”组织起来的校内学习平台。
- 它的核心不是单纯看课，而是“备课 -> 上课 -> 签到 -> 作品提交 -> 评价 -> 反思 -> 学期管理”这一整条信息科技课堂链路。
- 现有站点已经具备三种角色：学生、教师、管理员。
- 现有站点明显是单校模型，学校名、班级体系、学期配置、教师分配、机房等都默认只有一套。
- 你的新项目更像“校内产品升级 + 多校 SaaS 化”，我将它按“内部真实落地项目”来设计，而不是按演示型网站来设计。

## 3. 我已经确认的关键问题

### 3.1 视觉与体验问题

- 整体视觉停留在早期 WebForms 风格，灰色块和固定宽度布局过多
- 学生首页色彩单调，不符合初中生“更有趣、更自然、更愿意点进去”的审美
- 没有统一主题系统，无法切换学校主题或学生偏好主题
- 页面以桌面端固定布局为主，移动端和平板适配很弱
- 教师端和后台端风格几乎一致，没有把“教学操作”和“系统治理”做明显区分

### 3.2 架构与扩展问题

- 现有平台是典型 ASP.NET WebForms 模式，页面逻辑、状态、表单耦合较重
- 站点大量依赖 `__VIEWSTATE`，迁移维护成本高
- 单校模型写死在站点命名、班级列表、教师分配和系统设置里
- 多校情况下，现有“全校班级列表”思维无法直接扩展
- Excel 导入、数据备份、数据清理等都直接挂在后台页面按钮上，平台化治理能力不足

### 3.3 权限与安全问题

- 学号查询页会公开整班学生名单与学号入口，存在明显的隐私暴露风险
- 教师管理页直接显示教师明文密码，这是严重安全问题
- 教师端学生管理页可直接看到学生密码标题，同样不应继续保留
- 数据库备份与恢复、清空整班学生、历史清理等高风险操作直接暴露在 Web 管理页
- 教师端和后台端共用一个登录页，不利于权限隔离和行为审计

### 3.4 本轮不纳入处理的事项

- `teacher/systeminfo.aspx` 的现网异常本轮不作为重构判断依据处理
- 学生端真实登录落点已由用户明确为 `student/myinfo.aspx`，后续原型与实现应按这一入口组织学生首页

## 4. 新平台应该怎么“模仿”

这里的“模仿”我建议理解为：

- 业务流程模仿
- 角色结构模仿
- 核心模块模仿
- 教学组织方式模仿
- 不模仿旧 UI
- 不模仿旧安全模型
- 不模仿单校数据库思维

换句话说，新平台要保留它“非常懂信息科技课堂”的骨架，但必须重做它“老旧、单校、脆弱”的外壳和底座。

## 5. 新平台产品定位

平台名称暂定：信息科技学习综合门户平台

目标用户：

- 学生
- 教师
- 学校管理员
- 平台管理员

目标场景：

- 初中信息科技日常教学
- 多校统一部署
- 学校内网或教育云部署
- 课堂教学、作业作品、资源学习、成绩评价、学年管理一体化

核心价值：

- 对学生更好看、更清晰、更有参与感
- 对教师更顺手，减少重复性管理
- 对学校更安全，能做数据治理与权限治理
- 对多校部署更友好，支持学校独立配置和平台统一运营

## 7. 多学校模型设计

这是本次重构最关键的升级点。

### 7.1 推荐层级

- 平台
- 学校
- 校区
- 学年
- 学期
- 年级
- 班级

### 7.2 核心原则

- 所有业务数据必须带 `school_id`
- 允许每个学校独立配置名称、Logo、主题、学年、学期、登录页文案
- 教师默认只能看到所属学校和授权班级
- 学校管理员只能管理本校
- 平台管理员才能跨校查看和配置

### 7.3 为什么不能继续用旧模型

- 旧后台的“全校班级完整列表”在多校场景下会变成灾难
- 当前学期、下载限制、作品查看时间等不能再做成全平台唯一值
- 学生导入、教师分配、机房布置都必须先选学校，再做校内管理

## 8. 前端设计方向

### 8.1 技术栈

- Vue 3
- Vite
- TypeScript
- Vue Router
- Pinia
- Element Plus
- ECharts
- Sass

### 8.2 视觉方向

- 面向初中生，颜色要更丰富自然，不要廉价荧光色
- 登录页和学生端要更活泼，教师端和管理端要更专业，但仍保持统一品牌
- 全站做主题系统，至少支持 4 套主题

建议主题：

- 青柠森林
- 海盐天空
- 橙光操场
- 暖木书房

### 8.3 学生端设计原则

- 首页要像“学习乐园”，不是冷冰冰的表单页
- 当前课程、今日任务、我的作品、班级排行要用卡片化方式组织
- 课堂进度和作品状态要可视化
- 颜色、图标、留白、动效都要更友好

### 8.4 教师与后台设计原则

- 教师端强调高频操作效率
- 后台端强调结构清晰、风险提醒、批量处理
- 不能再把所有功能都堆到左侧一列灰色菜单里

### 8.5 临时设计系统基线

说明：

- 当前项目没有 `DESIGN.md`
- 在正式独立设计系统产出前，以下设计令牌与规则视为临时设计基线

#### 字体系统

- 学生端展示标题：`ZCOOL XiaoWei`
- 全站正文与控件：`HarmonyOS Sans SC`
- 数字、统计、英文标签：`IBM Plex Sans`

约束：

- 不使用默认系统字栈作为首选字体
- 全站最多使用 3 套字体角色
- 学生端可以在大标题上更有性格，教师端和后台端必须优先可读性

#### 色彩系统

- 基础中性色：暖灰 + 冷灰双系
- 学生端主色基调：自然明亮，不使用荧光感重的高饱和色
- 教师端主色基调：低饱和蓝绿灰，强调可读与状态感知
- 后台端主色基调：中性灰 + 明确风险色，不做花哨氛围背景

状态色：

- 成功：绿色系
- 警告：橙色系
- 错误：红色系
- 信息：蓝色系
- 正在进行：青色系

#### 间距系统

- 采用 4px 基础单位
- 核心布局间距使用：8 / 12 / 16 / 24 / 32 / 48 / 64
- 教师端与后台端以 16 / 24 为主
- 学生端卡片区允许更宽松的 24 / 32 / 48

#### 圆角与阴影

- 全站不使用统一夸张大圆角
- 小型控件：8px
- 卡片与容器：16px
- 大型展示区：24px 上限
- 阴影只用于层级，不用于装饰堆叠

#### 动效系统

- 页面加载：轻微上浮 + 渐显
- 列表进入：短距离 stagger
- 状态变化：颜色与透明度平滑过渡
- 学生端允许更明显的轻动效
- 教师端和后台端动效必须服从效率，不得打断操作

### 8.6 信息架构与布局规则

#### 学生端布局规则

- 首页不是传统门户，而是“今日学习驾驶舱”
- 首屏不超过 3 个高优先级区块
- 首屏必须先给出“现在该做什么”，而不是先给导航
- 卡片可以使用，但必须围绕任务和进度，不做装饰性卡片墙

#### 教师端布局规则

- 教师端核心页面不是 dashboard 拼贴，而是工作台
- 上课控制台必须同时支持：
- 选班
- 看状态
- 发起操作
- 查看求助
- 进入批改
- 不能让老师在一节课里跨多个页面来回跳

#### 后台端布局规则

- 后台页优先使用“主工作区 + 辅助上下文”结构
- 风险操作必须独立分组并加视觉警示
- 导入、迁移、清理、回滚等页面必须先展示当前上下文，再展示操作按钮

| 迁移兼容中心 | 上传/预检/执行进度条 | 无批次时显示“上传旧模板开始第一批迁移” | 明确字段错误、重复 ID、执行失败原因 | 批次完成并给出摘要 | 部分成功时必须显示失败明细与回滚入口 |
| 教师 AI 副驾 | “生成中”草稿态 | 暂无建议时展示触发入口 | 模型失败时显示可重试 + 不阻塞主流程 | 草稿生成成功，可编辑后采纳 | 结果可用但部分内容被过滤时显示说明 |

### 8.9 用户旅程与情绪设计

#### 学生旅程

| 步骤 | 用户动作 | 用户感受 | 设计支持 |
|------|----------|----------|----------|
| 1 | 打开登录页 | 想快速进入，不想被复杂表单吓住 | 登录页大动作优先，视觉友好 |
| 2 | 登录进入首页 | 想知道现在要做什么 | 当前课堂 Hero 放在第一屏 |
| 3 | 完成任务 | 希望有进度感 | 明确的完成状态与反馈 |
| 4 | 提交作品 | 怕提交失败 | 提交状态清晰，可重试 |

#### 教师旅程

| 步骤 | 用户动作 | 用户感受 | 设计支持 |
|------|----------|----------|----------|
| 1 | 登录进入控制台 | 需要立即进入工作状态 | 班级/课次/风险信息首屏可见 |
| 2 | 上课中监控全班 | 害怕漏掉异常学生 | 雷达区持续可见，优先显示异常 |
| 3 | 求助与批改 | 需要快速判断优先级 | 未处理任务按紧急度排序 |
| 4 | 使用 AI 草稿 | 既期待省时，又不完全信任 | 结果始终是草稿，不是自动替换 |

#### 管理员旅程

| 步骤 | 用户动作 | 用户感受 | 设计支持 |
|------|----------|----------|----------|
| 1 | 进入后台 | 害怕做错高风险操作 | 上下文与风险提醒先于按钮 |
| 2 | 上传迁移文件 | 担心格式错误 | 预检、预览、差异提示 |
| 3 | 执行批次 | 害怕部分失败不可恢复 | 批次状态机 + 回滚入口 |

### 8.10 AI Slop 防线

本项目明确禁止以下设计模式：

- 蓝紫渐变 SaaS 首页
- 三栏对称功能卡片作为主视觉
- 图标圆圈 + 标题 + 两行文案的模板化功能区
- 所有标题居中、所有内容等宽排列
- 为了“显得现代”堆大圆角、发光阴影、装饰性 blob

本项目应坚持：

- 首屏是一个完整构图，不是功能卡片堆砌
- 学生端像“任务舞台”，教师端像“课堂工作台”，后台像“治理控制台”
- 每个区域只有一个职责
- 页面一眼就能分清主任务、次任务和补充信息

### 8.11 响应式设计

#### 断点

- `>= 1440px`：宽屏工作台
- `1024px - 1439px`：标准桌面
- `768px - 1023px`：平板
- `390px - 767px`：手机

#### 响应规则

- 学生端在手机上优先保留当前任务与提交入口，隐藏低优先级信息
- 教师端手机不作为主上课设备，只保证可查看、可应急，不承诺完整高密度操作
- 教师端平板必须可完成基本课堂管理
- 后台端手机只提供只读和轻量审批，不提供高风险批量操作

### 8.12 可访问性要求

- 键盘可达：所有核心操作必须支持 Tab 顺序和焦点可见
- 最小点击区域：44px
- 颜色对比符合 WCAG AA
- 雷达状态不能只靠颜色区分，必须同时有图标/文案/形状差异
- 表格与高密度数据区必须支持屏幕阅读器语义
- 学生端任务状态需要图形化和文字双重表达

## 9. 后端设计方向

### 9.1 技术栈

- FastAPI
- SQLite3（当前本地开发数据库）
- SQLAlchemy 2.0
- Pydantic
- Alembic
- Redis
- MinIO 或兼容 S3 的对象存储

### 9.2 为什么推荐这套

- FastAPI 适合做清晰的前后端分离接口
- SQLite3 适合当前本地开发、快速启动和单机调试
- SQLAlchemy + Alembic 可以保证后续切换到 PostgreSQL、MySQL 等数据库时迁移成本可控
- Redis 可以承担登录态、缓存、异步任务协调
- 对象存储可以承接作品、资源、Excel 导入文件、备份文件

### 9.3 当前数据库策略

- 当前阶段先使用本地 `SQLite3`
- 数据访问统一通过 `SQLAlchemy ORM`，不要在业务代码里写死数据库方言
- 数据结构变更统一走 `Alembic`，不要手工改库
- 第一版表结构设计时避免过早依赖某个数据库特有能力
- 后续正式选型数据库时，再根据并发量、部署环境、备份策略和多校规模决定是否切换到更强的数据库

### 9.4 接口分层

- 认证与用户中心
- 学校与组织架构
- 教学课程与学案
- 课堂与签到
- 作品与评价
- 资源中心
- 数据导入导出
- 审计与运维

## 18. Office Hours 诊断

### 18.1 Demand Reality

我的判断：这个项目有真实需求，不是假想需求。

依据不是“这个方向看起来不错”，而是：

- 旧系统里存在真实的教师、学生、学案、签到、作品、学年数据
- 页面中的日期和记录显示它正在被持续使用
- 教师端和后台端功能都围绕高频课堂操作展开，而不是展示型页面

这意味着需求已经被验证。真正的问题不是“要不要做”，而是“新系统先替代旧系统的哪一段，才能最先产生价值并降低迁移阻力”。

### 18.2 Status Quo

我的判断：你的真正竞争对手不是别的教育平台，而是“旧站 + Excel + 老师现有操作习惯”。

当前状态的真实工作流大概是：

- 教师在旧后台里备课、发布、上课、看签到、看作品
- 管理员通过 Excel 导入学生、升班、分班
- 学校在单校后台里维护教师、班级、机房、学期

所以新系统如果只是“更现代”，但不能在这些日常动作上更顺手，它就赢不了旧系统。

### 18.3 Desperate Specificity

我认为这个系统最痛的核心用户不是“学生”这个大类，而是下面三类具体人：

- 信息科技教师：上课前要快速选班、发课、看签到、收作品、批改
- 教研组长或骨干教师：要维护课程资源和教学质量，又不想被老系统拖慢
- 学校管理员或信息中心老师：要导入学生、分配教师、管学期、管班级、处理数据

谁最容易推动项目？

- 不是学生
- 也不是平台开发者
- 而是“每天都在旧系统里做重复操作、又最清楚旧痛点的教师和校内管理员”

### 18.4 Narrowest Wedge

我的判断非常明确：

- “多学校综合门户平台”是正确愿景
- 但“第一版就做完整多校大平台并全面替换所有模块”是错误切口

最小可落地切口应该是：

- 架构按多学校设计
- 上线按单校试点推进
- 功能先打通一条最核心教学闭环

我建议的一期最窄闭环：

- 学生登录
- 教师登录
- 管理员登录
- 学案管理
- 上课控制台
- 课堂签到
- 作品提交与批改
- 学生管理
- 班级/教师/学期配置

也就是说：

- 多学校是底座能力，不是一期宣传口号
- 单校试点才是一期产品切口

### 18.5 Observation & Surprise

基于现有站点结构，我有一个很重要的判断：

- 这个产品最值钱的地方，不是“内容浏览”
- 而是“课堂现场控制能力”

为什么这么说：

- 旧教师端最重的功能是上课、签到、作品、学生、学案，不是首页宣传或内容社区
- 后台最重的是导入、分班、升班、权限、机房，不是营销能力
- 学生端入口虽然简单，但其存在价值是低门槛进入课堂任务，而不是复杂的个性化学习空间

这意味着新平台的产品中心，不该是“门户首页设计得多漂亮”，而应该是：

- 让老师在 40 分钟课堂里少点几次、多看见关键状态、少出一次错

这条判断会直接影响你后续的 UI 和架构优先级。

### 18.6 Future-Fit

如果这个项目被包装成：

- “一个更现代、更漂亮的校园学习网站”

那它很容易在资源紧张时被砍掉。

但如果它被定义成：

- “面向信息科技课堂的教学操作系统，降低教师重复操作，支撑多学校复制”

它就更容易活下来，因为它解决的是日常运营问题，不只是视觉升级。

所以我的判断是：

- 这个项目必须被定义为“教学工作流升级”
- 而不是“旧站前端翻新”

2. 一期必须替代的旧系统流程是哪 6 到 8 个
3. 一期是否迁移旧数据，还是新旧并行一段时间
4. 一期试点学校或试点校区是谁

我的推荐答案是：

- 一期按“单校试点、底层多校”推进
- 一期先替代登录、学案、上课、签到、作品、学生、学期、班级分配
- 一期同时建设迁移兼容层，优先兼容旧 Excel 模板、旧账号导入和旧资源映射
- 课堂实时雷达与教师 AI 副驾纳入一期计划，但按后置子阶段交付
- 历史数据迁移不要一开始全做，先迁核心组织数据和必要课程数据
- 在真实学校场景里先跑通一学期的闭环

### 22.6 Landscape Synthesis

#### Layer 1: Tried and True

K-12 LMS 和校务平台的成熟共识已经很明确：

- 核心平台不能只是内容仓库，必须连接用户、权限、内容、分析、沟通和第三方系统
- 安全与权限、集成能力、内容管理、进度分析是基础能力
- 真正能长期留下来的系统，不是 feature 多，而是“把老师已有工作流整合进一个更少登录、更少跳转的环境”

#### Layer 2: Current Signals

从 2025-2026 的公开资料看，现在最强的共识有四条：

- 学校想要更少而不是更多的工具，重点是整合与单点登录
- AI 正在成为课堂平台的默认组成，但学校更在意 guardrails、权限和一致规则
- 数据隐私、角色权限、恢复能力已经从 IT 议题变成教学连续性议题
- 可访问性、辅助功能和异质化学习支持正在从加分项变成基线

#### Layer 3: First Principles

我认为传统 LMS 思维在这个项目里有一个错位：

- 它默认“课程内容分发”是核心
- 但你的旧系统已经证明，对信息科技课堂来说，真正的核心是“教师在课中如何管理一群学生、任务、作品和时间”

这就是这个项目潜在的 eureka：

- 你不该做“又一个学校门户”
- 你应该做“信息科技课堂的操作系统”

#### 24.2.4 Search Check

基于官方资料的工程判断：

- **[Layer 1]** FastAPI 已经提供官方 `WebSocket` 支持，也在 `0.135.0` 起提供了官方 SSE 支持，实时雷达不需要自造长轮询协议或自定义流式框架
- **[Layer 1]** SQLite 和 SQLAlchemy 官方文档都明确提醒：SQLite 不适合高写并发；即使 WAL 能改善并发，也仍然存在单写者约束，因此只能用于本地开发，不应作为实时雷达试点环境数据库

### 24.3 Architecture Review

#### Issue 1: 课堂实时雷达的数据通道没有锁定

当前问题：

- 文档写了“实时雷达”，但没有明确学生端如何上报、服务端如何聚合、教师端如何订阅

风险：

- 如果直接上全双工 WebSocket，会把认证、断线重连、多标签页、移动网络抖动、服务端 fan-out 一次性全引进来
- 如果继续停留在“以后再说”，实现时会出现前后端各自理解不同、难测试、难压测的问题

工程建议：

- 第一版采用“学生端 HTTP 心跳上报 + 教师端 SSE 订阅更新”的组合
- 只在确实需要双向实时交互时再升级为 WebSocket

原因：

- 教师端主要是接收状态，不是持续发送复杂指令
- SSE 在浏览器和 FastAPI 里已经有官方支持，复杂度和观测难度都低于 WebSocket

ASCII 数据流图：

```text
student browser
    │
    ├─ POST /presence/heartbeat
    ├─ POST /help-requests
    └─ POST /task-status
            │
            v
      FastAPI teaching_core
            │
            ├─ validate session + school scope
            ├─ write raw event
            └─ update live_presence_snapshot
                        │
                        v
             GET /teacher/session/{id}/radar/stream
                        │
                        v
                teacher dashboard (SSE)
```

#### Issue 2: 多学校权限隔离还停留在数据字段层

当前问题：

- 现在文档写了“所有业务数据都带 `school_id`”，但这只是数据结构，不是执行保障

风险：

- 最常见的越权不是 schema 没字段，而是查询漏过滤、导出接口漏校验、后台管理员跨校看见数据

工程建议：

- 引入 request-scoped `SchoolContext/TenantContext`
- 所有查询入口必须显式带 school scope
- 管理员角色分成本校管理员和平台管理员两类，导出与批处理权限单独判定

ASCII 请求边界图：

```text
request
  │
  ├─ auth middleware
  │      └─ resolve user + roles
  │
  ├─ school context resolver
  │      └─ active_school_id / allowed_school_ids
  │
  ├─ route handler
  │      └─ forbid cross-school access unless platform-admin
  │
  └─ repository/query layer
         └─ every query scoped by school context
```

#### Issue 3: 迁移兼容层缺少状态机

当前问题：

- 计划已经写了兼容旧 Excel、旧账号、旧资源映射，但没有写迁移作业的生命周期

风险：

- 一旦导入失败、重复导入、部分成功、字段不一致，没有状态机就无法重试、回滚、对账

工程建议：

- 每个迁移批次都必须有明确状态：
- `draft`
- `validated`
- `previewed`
- `executing`
- `partially_failed`
- `completed`
- `rolled_back`

ASCII 状态机：

```text
draft
  │ validate
  v
validated
  │ preview
  v
previewed
  │ execute
  v
executing
  ├─ success ----------> completed
  ├─ partial failure --> partially_failed
  └─ manual rollback --> rolled_back
```

补充要求：

- 所有导入必须支持 dry-run
- 所有批次必须有错误明细和人工修复入口
- 所有映射表必须是幂等可重放的

#### Issue 4: 教师 AI 副驾缺少信任边界

当前问题：

- 计划写了生成课堂建议、量规初稿、分层练习和作品反馈草稿，但没有定义 AI 什么时候可自动通过，什么时候必须人工审核

风险：

- AI 输出错误、偏题、泄露敏感信息、对学生做不一致评价
- 服务不可用时拖垮主流程

工程建议：

- 第一版 AI 全部做“建议草稿”，不允许直接生效
- 所有 AI 输出都要落审计记录，关联教师、学案、上下文、模型版本
- AI 调用失败不能阻塞主课堂链路

AI 边界图：

```text
teacher action
   │
   ├─ request AI suggestion
   │        │
   │        ├─ build prompt from sanitized context
   │        ├─ call model provider
   │        ├─ save draft + metadata
   │        └─ return editable draft
   │
   └─ teacher reviews
            ├─ accept
            ├─ edit then accept
            └─ reject
```

#### Issue 5: 发布与运行环境没有写清

风险：

- 做出来的代码只能本机跑，无法让学校稳定试点

工程建议：

- 至少明确：
- `frontend` 打包方式
- `backend` 启动方式
- 静态资源与对象存储策略
- 本地开发环境
- 单校试点环境

### 24.6 Performance Review

#### 24.6.1 SQLite 只能用于本地开发

根据 SQLite 和 SQLAlchemy 官方文档：

- SQLite 不适合高写并发
- 即使 WAL 能提升读写并发，也仍然存在单写者约束

工程结论：

- `SQLite3` 可用于当前本地开发
- 但在进入单校试点前，必须切换到正式数据库
- 尤其是课堂实时雷达这种持续写入型场景，不能拿 SQLite 作为试点环境数据库

#### 24.6.2 实时雷达不能每次都扫原始事件表

如果教师端每次刷新都直接从 `live_session_events` 聚合：

- 会产生明显的热点查询
- 会让性能和实现复杂度一起变差

建议：

- 原始事件表只做审计和回放
- 单独维护 `live_presence_states` 作为当前快照

#### 24.6.3 AI 不能阻塞主请求

教师 AI 副驾如果同步等待模型返回：

- 高延迟会直接拖慢页面
- 外部模型波动会拖垮用户体验

建议：

- AI 生成统一走异步任务
- 前端看到“生成中/已完成/失败可重试”的草稿状态

### 24.7 推荐锁死的工程决策

1. 课堂实时雷达：
推荐采用 `HTTP heartbeat + SSE stream` 作为第一版基线

2. 多学校隔离：
推荐采用 request-scoped school context + query 层强制过滤

3. 迁移兼容层：
推荐采用批次状态机 + dry-run + rollback + immutable mapping

4. 教师 AI 副驾：
推荐采用 `teacher-in-the-loop draft workflow`，禁止自动生效

5. 数据库策略：
推荐 `SQLite3` 只用于本地开发，试点前切正式数据库

### 24.8 工程经理结论

这份计划可以开工，但现在还不能直接开工。

还差的不是“更多功能”，而是五个必须锁死的工程约束：

- 雷达协议
- 租户隔离执行方式
- 迁移状态机
- AI 信任边界
- 测试与发布约束

只要这五个点补进计划，这份方案就能从“方向正确”升级到“工程可执行”。
